title: "Name the lane, not the model"
date: 2026-06-10

summary: "Inference deployments should be named for the role they serve, never the model they happen to be running. A small naming decision that keeps paying for itself."
---

My cluster's first vLLM deployments were named after the models they served:
`vllm-cora` for the general-purpose lane, `vllm-codestral` for inline
completion. One of those names turned out to be a quiet liability, and the
difference between them became a rule I now apply to every inference
deployment: **name the lane, not the model.**

## How a deployment name starts lying

`vllm-cora` happened to be role-named — "cora" is the assistant lane, and the
lane survives any model rotation. `vllm-codestral` embedded the model name,
which meant:

1. **It lies on rotation.** The moment the inline-completion lane swaps to a
   different model, the deployment name is misleading until someone renames
   it. This isn't hypothetical — that deployment later spent two weeks named
   `vllm-codestral` while serving (or trying to serve) something that was
   never going to be Codestral again.
2. **It forces churn at every model swap.** Rename the deployment, update the
   gateway's `model_name`, update the Grafana dashboards, update operator
   muscle memory. Every rotation pays the cost again.
3. **It doesn't compose.** A second inline-completion specialist — different
   model, different card — has no naming room without colliding.

## The rule

Inference deployments are **role-named**, never model-named. The role
describes the SLA / use-case lane: general-purpose primary, code review,
large-MoE batch, multimodal vision. Models rotate within a lane without any
deployment-side change.

| Role | Deployment | Gateway name |
|---|---|---|
| Primary general-purpose | `vllm-cora` | `vllm/cora` |
| Reviewer (T0/T1) | `vllm-review` | `vllm/review` |
| Large MoE batch | `vllm-large` | `vllm/large` |
| Vision multimodal | `vllm-vision` | `vllm/vision` |

The actual served model lives where it belongs: in the deployment YAML's
`--model` argument and comment header, in the model registry, and in the
gateway config's paired `model:` line.

The detail that makes this airtight: vLLM's `--served-model-name` is *also*
role-named. The served name becomes the `model:` field in OpenAI-compatible
API requests, and the gateway forwards using it. Keeping it role-aligned
means the deployment name, the gateway alias, and the wire-level model field
are a single role label end-to-end. Less surface for mistakes.

## The alternatives, and why they lose

**Hardware-anchored** (`vllm-a6000`, `vllm-3090`): describes the card, hides
both model and role, and breaks the day the card moves to another host —
which, on this cluster, it did.

**Model-as-name** (`vllm-qwen3-27b`): maximally explicit, maximally churny.
Every rotation is a deployment rename, and rotations are the *point* of an
inference platform.

**Tier-only abstractions** (`inference-flagship`): more abstract but less
greppable, and loses the engine prefix that's already burned into dashboards
and runbooks.

## Paying the migration cost once

Renaming the model-named lane was a ~20-file sweep: deployment, service,
configmap, gateway model name and alias map, ServiceMonitor selector,
exporter comments, cross-reference comments in four sibling apps, two
dashboard descriptions, and a documentation example. That's exactly the
churn the rule exists to avoid — paid once, deliberately, instead of at
every future rotation in the lane.

The second application of the rule was cheaper in the way that proves the
point: the rename rode along with a model rotation that was happening
anyway, because the rotation is precisely the moment a model-name lie
surfaces.

## The wider shape

This is a tiny decision, but it's the same decision as "don't name your
servers after the application they run" and "don't name your branch after
the week you cut it" — bind names to the stable thing, not the current
occupant. On an inference platform the stable thing is the lane: its SLA,
its consumers, its card class. The model is a tenant.

If you run more than one model server, the test is cheap: look at each
deployment name and ask what happens to it on the next rotation. If the
answer is "someone has to rename it, and until they do it lies" — name the
lane instead.

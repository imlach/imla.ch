This is a public write-up of an internal post-mortem from my homelab
Kubernetes cluster — a 14-node k3s fleet on Proxmox that runs real studio
workloads (media services, CI runners, an ML inference stack) and doubles as
the place I test ideas before they harden. Names and numbers are real;
nothing here was customer-facing.

The short version: a control-plane migration handled everything *inside* its
scope flawlessly, and the cluster still degraded for hours — because three
static configs *outside* that scope each held a hardcoded copy of the old
control-plane IPs. A host OOM arrived in the same window, just to keep
things interesting.

## The setup

The migration split the k3s control plane from the worker nodes: three new
dedicated CP-only VMs at `10.0.31.40–42`, with the original mixed nodes at
`10.0.31.1–3` demoted to workers. The demote playbook handled the etcd
member list correctly — promotion, demotion, quorum, all clean.

Three hours later, services started failing cluster-wide.

The delay is the interesting part. Nothing broke at demote time because
in-flight Cilium pods kept their cached apiserver connections, and those
cached connections kept working. The failure was armed, not triggered. The
trigger was the next pod restart.

## The cascade, in order of discovery

**Root cause #1 — the CNI's apiserver address.** Cilium's Helm values had
`k8sServiceHost: "10.0.31.1"` hardcoded — a direct control-plane IP rather
than the VIP, deliberately, to avoid the CNI-needs-the-LB-to-reach-the-apiserver
bootstrap circle. Correct in spirit; tied to a node that no longer ran an
apiserver. Every *fresh* Cilium pod read that env var, retried for 60
seconds, and crash-looped. With the operator down, the BGP-advertised VIP
stopped being advertised at all, and the cascade widened as anything that
restarted got stuck `ContainerCreating`.

The log line was the smoking gun:
`Unable to contact k8s api-server ipAddr=https://10.0.31.1:6443 error=connection refused`.
Live-patching the env var on the DaemonSet and operator recovered most of
the cluster in ten minutes.

**Root cause #2 — the router's BGP neighbor list.** The VIP stayed down.
The gateway's FRR config enumerated BGP neighbors `10.0.31.1` through `.5` —
the new control-plane nodes weren't in the list, so the router refused their
TCP/179. This one had been masked by #1: while the Cilium operator (which
owns BGP) was down, there was no peering attempt to observe failing.

**Root cause #3 — a manually-managed EndpointSlice.** BGP established, routes
advertised — and the VIP answered with TCP resets. The API-server LoadBalancer
Service was backed by a hand-managed EndpointSlice (GitOps excludes endpoint
resources, so a playbook owns it), and it still listed `10.0.31.1–3:6443`.
Nothing was listening there. Patch, recovered.

Three fixes, strictly sequential — each invisible until the previous one was
applied. Fixing only #1 restores the cluster but not the VIP. Fixing #2
establishes BGP but traffic dies at the stale endpoints. Only with all three
does `kubectl` via the VIP work again.

**Root cause #4 — independent, concurrent.** One Proxmox host was
over-committed by design: 49 GiB of VM allocation on 46 GiB usable. That
arithmetic only works while no VM hits its configured ceiling, and during
the incident window the largest VM hit its workload peak. The kernel OOM
killer chose the biggest qemu process — a 33 GiB GPU worker — and shot it.
Unrelated to the migration; it would have happened under any sustained load.
It just chose its moment.

## What the migration playbook missed

Nothing it was asked to do. It handled etcd membership, service migration,
taint removal, GPU passthrough preservation — the complete checklist of
"remove a node from the etcd cluster". What it had no awareness of was every
*other* place in the infrastructure that had ever written down a
control-plane IP:

- the CNI's bootstrap apiserver address,
- the router's BGP neighbor list,
- a manually-managed EndpointSlice.

Three different config surfaces, three different owners (Helm values, a
router config file, a playbook-managed resource), one shared property: a
static copy of `10.0.31.[123]` that nothing audited.

## The mechanical fix

The lesson isn't "be more careful" — it's that **IP migrations need a
pre-flight grep, enforced by tooling**. The durable change to come out of
this incident is an audit playbook that runs before any migration that
retires, renames, or repoints a node identity: search the entire IaC tree
*and* live cluster resources for the affected identifiers, and fail the run
if any reference survives that isn't explicitly allow-listed. It would have
caught all three root causes before they were armed.

(It also reports the surfaces it *can't* reach — the router UI, external
dashboards — as a manual checklist, which is the honest version of
automation: do what you can, enumerate what you can't.)

Two more lessons earned their keep:

**"Tight by design" is fragile.** The over-committed host was visible in the
inventory and even described as tight in the decision record that created
it. KVM hard-caps each VM at its configured RAM, so an over-committed host
depends on *workload behaviour* to stay healthy — which is not a property of
the host. Either size the VMs to fit, or add RAM. We consolidated: fewer,
bigger VMs per host, with measured headroom.

**Pointing at a different single IP is not a fix.** Each of the three root
causes was repaired by replacing one hardcoded IP with another. That clears
the symptom and preserves the architecture that produced it — lose the new
first control-plane node and root cause #1 recurs verbatim. The real
hardening (multi-endpoint client config, neighbor health rather than
neighbor lists, service-managed endpoints) is tracked separately, and being
honest that it's *not done yet* is part of what a post-mortem is for.

## What went well

Worth recording, because post-mortems that only list failures train you to
hide incidents:

- **Logs pointed at the answer.** The first root cause was identified within
  minutes of actually looking, because the failing component printed the
  stale IP it was dialling.
- **A fallback access path existed.** With the VIP down, direct per-node
  apiserver access kept the operator inside the problem instead of locked
  out of it.
- **Live patches and GitOps converged.** Each emergency `kubectl` fix was
  followed by the matching IaC change before the reconciler could disagree
  with it. No fix got silently reverted.
- **The new architecture held.** The CP/worker split this migration delivered
  was itself the fix for an older class of instability — and that fix
  worked. This incident was configuration drift layered on top of a
  genuinely better foundation, which is annoying, but it's the *right kind*
  of annoying.

## The takeaway

A migration is correct when nothing in the system still believes the old
topology — not when the playbook exits zero. The etcd member list is just
the copy of the topology the migration tool happens to know about. The
others are in a Helm values file, a router config, and an EndpointSlice
some playbook wrote eight weeks ago.

Grep for your old IPs. Mechanically, before every migration, with a tool
that fails loudly. The three hours of cached-connection grace period is the
universe offering you a window to do it in.
